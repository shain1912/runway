## ㄱ

| 용어 | 설명 |
|------|------|
| **거버넌스** (Governance) | 조직의 AI 운영을 체계적으로 관리하기 위한 정책과 통제 체계입니다. 모델 배포 승인 프로세스, 리소스 사용 정책, 데이터 접근 제어 등을 포함합니다. 안전하고 효율적인 AI 운영을 위한 필수 요소입니다. |
| **고가용성** (High Availability) | 시스템이 장애 상황에서도 중단 없이 지속적으로 서비스를 제공할 수 있는 특성입니다. 다중화, 자동 복구, 로드 밸런싱 등의 기술을 통해 구현됩니다. 프로덕션 환경에서 중요한 시스템 요구사항입니다. |
| **공유 메모리** (Shared Memory) | GPU 워크로드에서 여러 프로세스가 데이터를 공유하기 위해 사용하는 메모리 영역입니다. IPC(Inter-Process Communication)를 통해 프로세스 간 빠른 데이터 교환을 가능하게 합니다. 특히 멀티 GPU 환경에서 모델 학습 시 성능 향상에 중요한 역할을 합니다. |
| **관리자** (Admin) | 프로젝트 또는 워크스페이스에서 모든 권한을 가진 역할입니다. 리소스 관리, 멤버 초대 및 권한 부여, 설정 변경 등 모든 작업을 수행할 수 있습니다. 일반적으로 프로젝트나 워크스페이스를 생성한 사용자에게 자동으로 부여됩니다. |
| **권한** (Permission) | 사용자가 특정 작업을 수행할 수 있도록 부여되는 허가입니다. Runway에서는 역할(Role)을 통해 권한을 그룹화하여 관리합니다. 읽기, 쓰기, 실행, 삭제 등 다양한 수준의 권한이 존재합니다. |

## ㄴ

| 용어 | 설명 |
|------|------|
| **네임스페이스** (Namespace) | Kubernetes에서 리소스를 논리적으로 분리하는 단위입니다. Runway에서는 프로젝트마다 고유한 네임스페이스가 생성되어 프로젝트 간 리소스와 데이터가 격리됩니다. 멀티 테넌트 환경에서 보안과 독립성을 보장합니다. |
| **노드** (Node) | Kubernetes 클러스터를 구성하는 물리적 또는 가상의 워커 머신입니다. CPU, 메모리, GPU 등의 컴퓨팅 리소스를 제공하며, 여러 Pod를 실행할 수 있습니다. 클러스터는 여러 노드로 구성되어 워크로드를 분산 처리합니다. |

## ㄷ

| 용어 | 설명 |
|------|------|
| **데이터셋** (Dataset) | 모델 학습, 검증, 평가에 사용되는 구조화된 데이터 집합입니다. Runway에서는 데이터셋을 버전 관리하고 프로젝트 간 공유할 수 있습니다. 재현 가능한 ML 실험을 위해 데이터셋 추적성이 중요합니다. |
| **동기** (Synchronous) | 작업을 수행한 후 응답을 기다린 다음 다음 작업을 진행하는 순차적 처리 방식입니다. 작업 순서가 보장되며 결과를 즉시 확인할 수 있습니다. 비동기 방식에 비해 처리 시간이 길어질 수 있습니다. |
| **동기화** (Synchronization) | Git 리포지토리의 설정과 실제 클러스터 상태를 일치시키는 작업입니다. ArgoCD 같은 GitOps 도구가 주기적으로 차이를 감지하고 자동으로 동기화합니다. 인프라를 항상 최신 상태로 유지합니다. |
| **동적 배칭** (Dynamic Batching) | 여러 개의 추론 요청을 실시간으로 모아서 한 번에 처리하는 최적화 기법입니다. GPU 연산 효율을 극대화하여 처리량을 크게 증가시킵니다. 레이턴시가 약간 증가하지만 전체 시스템 성능이 향상됩니다. |

## ㄹ

| 용어 | 설명 |
|------|------|
| **라이브러리** (Library) | 재사용 가능한 코드, 함수, 클래스를 모아놓은 패키지입니다. 개발자는 라이브러리를 사용하여 반복적인 작업을 줄이고 개발 생산성을 높일 수 있습니다. PyTorch, TensorFlow 같은 ML 프레임워크도 라이브러리의 일종입니다. |
| **레이턴시** (Latency) | 요청을 보낸 시점부터 응답을 받을 때까지의 지연 시간입니다. 모델 서빙에서 중요한 성능 지표로, 낮은 레이턴시는 빠른 응답 속도를 의미합니다. 실시간 서비스에서는 레이턴시가 사용자 경험에 직접적인 영향을 미칩니다. |
| **로깅** (Logging) | 시스템의 동작, 이벤트, 오류 등을 기록하는 작업입니다. 로그는 문제 진단, 성능 분석, 보안 감사에 활용됩니다. 효과적인 로깅 전략은 시스템 운영의 필수 요소입니다. |
| **롤백** (Rollback) | 배포 후 문제가 발생했을 때 이전에 정상 동작하던 버전으로 되돌리는 작업입니다. 트래픽 비율 조정이나 배포 설정 변경을 통해 빠르게 서비스를 복구할 수 있습니다. 안정적인 서비스 운영을 위한 필수 기능입니다. |
| **롤아웃** (Rollout) | 새로운 버전의 모델이나 애플리케이션을 점진적으로 배포하는 전략입니다. 카나리 배포, 블루-그린 배포 등 다양한 방식이 있으며, 전체 트래픽에 대한 영향을 최소화합니다. 문제 발생 시 빠르게 이전 버전으로 롤백할 수 있습니다. |
| **리소스 사용률** (Resource Utilization) | 전체 용량 대비 실제로 사용 중인 리소스의 비율입니다. 클러스터 효율성을 평가하는 핵심 메트릭으로, 너무 낮으면 리소스 낭비, 너무 높으면 성능 저하를 의미합니다. 적정 사용률 유지가 중요합니다. |
| **리소스 할당량** (Resource Quota) | 워크스페이스나 프로젝트에 할당된 최대 리소스 한도입니다. CPU, 메모리, GPU 등 각 리소스 유형별로 할당량을 설정할 수 있습니다. 할당량을 통해 팀 간 리소스 사용을 공정하게 관리하고 과다 사용을 방지합니다. |

## ㅁ

| 용어 | 설명 |
|------|------|
| **메모리** (Memory) | 애플리케이션 실행에 필요한 RAM(Random Access Memory) 자원입니다. 워크로드 생성 시 필요한 메모리 양을 지정하며, 부족하면 OOM(Out Of Memory) 오류가 발생합니다. 적절한 메모리 할당은 안정적인 애플리케이션 실행의 핵심입니다. |
| **메모리 맵** (Memory Map) | 파일 내용을 프로세스의 메모리 주소 공간에 직접 매핑하는 기법입니다. 대용량 파일을 효율적으로 읽거나 프로세스 간 데이터 공유에 사용됩니다. 일반 파일 I/O에 비해 성능이 우수합니다. |
| **메트릭** (Metric) | 시스템의 성능과 상태를 정량적으로 측정하는 지표입니다. CPU 사용률, 메모리 사용량, 처리량, 레이턴시 등이 대표적인 메트릭입니다. 메트릭을 통해 시스템 건강 상태를 파악하고 최적화 방향을 결정합니다. |
| **멤버** (Member) | 프로젝트에서 일반적인 작업 권한을 가진 구성원 역할입니다. 애플리케이션 생성, 모델 학습, 데이터셋 관리 등의 작업을 수행할 수 있습니다. 관리자에 비해 제한된 권한을 가지며, 프로젝트 설정이나 멤버 관리는 수행할 수 없습니다. |
| **모니터링** (Monitoring) | 시스템의 리소스 사용량, 성능 메트릭, 상태를 실시간으로 추적하는 기능입니다. CPU, 메모리, GPU 사용률, 요청 수, 응답 시간 등을 시각화합니다. 문제를 조기에 발견하고 대응하는 데 필수적입니다. |
| **모델 경로** (Model Path) | 프로젝트 볼륨 내에서 모델 아티팩트가 저장된 디렉토리 경로입니다. 모델 배포 시 이 경로를 지정하여 서빙 런타임이 모델 파일을 로드합니다. 일반적으로 /models/모델명/버전 형식으로 구성됩니다. |
| **모델 배포** (Model Deployment) | 학습된 모델을 엔드포인트에서 실행 가능한 형태로 배포하는 단계입니다. 모델 아티팩트, 서빙 런타임, 리소스 설정을 지정하여 배포를 생성합니다. 하나의 엔드포인트에 여러 모델 배포를 구성하여 A/B 테스트나 트래픽 분배를 수행할 수 있습니다. |
| **모델 서빙** (Model Serving) | 학습된 모델을 외부에서 호출 가능한 서비스로 제공하는 전체 프로세스입니다. 엔드포인트 생성, 모델 배포, 트래픽 관리, 모니터링을 포함합니다. Runway는 다양한 서빙 런타임을 지원하여 최적의 성능을 제공합니다. |

## ㅂ

| 용어 | 설명 |
|------|------|
| **바이너리** (Binary) | 컴퓨터가 직접 실행할 수 있는 이진 형식의 파일입니다. 소스 코드를 컴파일하여 생성되며, 사람이 읽을 수 없지만 실행 속도가 빠릅니다. 배포 시 소스 코드 노출을 방지하는 효과도 있습니다. |
| **배치 서빙** (Batch Serving) | 대량의 데이터를 한 번에 입력받아 일괄 처리하는 추론 방식입니다. 실시간 응답이 필요하지 않은 경우 효율적이며, 처리량(throughput)을 최대화할 수 있습니다. 야간 배치 작업이나 대규모 데이터 분석에 적합합니다. |
| **배포** (Deployment) | 애플리케이션이나 모델을 클러스터에 실행 가능한 상태로 만드는 작업입니다. 컨테이너 이미지, 리소스 설정, 환경 변수 등을 지정하여 워크로드를 생성하고 실행합니다. Runway에서는 No-Code 방식으로 간편하게 배포할 수 있습니다. |
| **배포 해제** (Undeploy) | 실행 중인 애플리케이션이나 모델을 중지하고 할당된 리소스를 반환하는 작업입니다. 배포 해제 후에도 설정은 보존되어 나중에 다시 배포할 수 있습니다. 비용 절감과 리소스 효율화를 위해 사용됩니다. |
| **베이스 도메인** (Base Domain) | Runway 플랫폼의 기본 도메인 주소입니다. 모든 애플리케이션과 엔드포인트의 URL이 이 베이스 도메인을 기반으로 생성됩니다. 예를 들어 runway.example.com이 베이스 도메인이면 앱 URL은 myapp.project.ws.runway.example.com 형식입니다. |
| **벡터 데이터베이스** (Vector Database) | 고차원 벡터 데이터를 저장하고 유사도 검색을 지원하는 특수 목적 데이터베이스입니다. 임베딩 벡터를 효율적으로 인덱싱하여 밀리초 단위로 가장 유사한 벡터를 찾습니다. RAG 시스템과 추천 엔진의 핵심 인프라입니다. |
| **벡터 임베딩** (Vector Embedding) | 텍스트, 이미지 등의 데이터를 고차원 벡터 공간에 표현하는 기법입니다. 의미가 유사한 데이터는 벡터 공간에서 가까운 위치에 매핑됩니다. RAG와 유사도 검색의 핵심 기술입니다. |
| **벤치마크** (Benchmark) | 모델의 성능을 객관적으로 평가하기 위한 표준화된 테스트입니다. 정확도, 처리 속도, 리소스 사용량 등 다양한 측면을 측정합니다. 서로 다른 모델이나 최적화 기법을 비교하는 데 사용됩니다. |
| **복제본** (Replica) | 동일한 애플리케이션이나 모델을 여러 개 실행하여 부하 분산과 고가용성을 확보하는 방식입니다. 하나의 인스턴스에 문제가 생겨도 다른 복제본이 계속 서비스를 제공합니다. Kubernetes의 ReplicaSet으로 관리됩니다. |
| **볼륨** (Volume) | 애플리케이션이 생성하고 사용하는 데이터를 영구적으로 보관하기 위한 저장 공간입니다. Pod가 삭제되어도 데이터가 유지되며, 여러 Pod가 동일한 볼륨을 공유할 수 있습니다. PVC, Shared PVC 등 다양한 볼륨 유형을 지원합니다. |
| **분산 시스템** (Distributed System) | 여러 컴퓨팅 노드에 걸쳐 작업을 분산하여 처리하는 시스템입니다. 확장성과 안정성이 뛰어나며, 단일 노드의 성능 한계를 극복할 수 있습니다. Kubernetes가 대표적인 분산 시스템 플랫폼입니다. |
| **뷰어** (Viewer) | 조회 권한만 가진 제한된 역할입니다. 프로젝트의 정보를 확인할 수 있지만 생성, 수정, 삭제 작업은 수행할 수 없습니다. 외부 이해관계자나 감사 목적으로 프로젝트를 공유할 때 유용합니다. |
| **비동기** (Asynchronous) | 작업을 시작한 후 응답을 기다리지 않고 즉시 다음 작업을 진행하는 처리 방식입니다. 여러 작업을 동시에 수행할 수 있어 효율적이지만, 작업 순서가 보장되지 않습니다. 콜백이나 Promise를 통해 완료 시점을 처리합니다. |

## ㅅ

| 용어 | 설명 |
|------|------|
| **사용자 정의 역할** (Custom Role) | 관리자가 특정 권한 조합을 선택하여 생성하는 커스텀 역할입니다. 기본 제공되는 Admin, Member, Viewer 외에 조직의 필요에 맞는 세부적인 권한 제어가 가능합니다. 최소 권한 원칙을 구현하는 데 유용합니다. |
| **섀도우 테스트** (Shadow Test) | 프로덕션 트래픽의 복사본을 새로운 모델로 전송하여 실제 영향 없이 검증하는 비동기 테스트 방식입니다. 실제 사용자에게는 기존 모델의 응답만 반환하고, 새 모델의 결과는 비교 분석용으로만 사용합니다. A/B 테스트 전에 안전하게 모델을 평가할 수 있습니다. |
| **서빙 런타임** (Serving Runtime) | 모델 추론을 실제로 실행하는 런타임 환경입니다. vLLM, TEI, Triton, llama.cpp 등 다양한 런타임이 있으며, 각각 특정 모델 유형이나 하드웨어에 최적화되어 있습니다. 적절한 런타임 선택이 추론 성능을 크게 좌우합니다. |
| **선언적 배포** (Declarative Deployment) | 원하는 최종 상태를 선언하면 시스템이 자동으로 현재 상태를 목표 상태로 동기화하는 배포 방식입니다. Kubernetes와 GitOps의 핵심 원칙으로 명령형 배포보다 안정적이고 예측 가능합니다. |
| **스케줄링** (Scheduling) | 작업을 특정 시간에 자동으로 실행하거나 특정 조건이 만족될 때 트리거하는 기능입니다. 주기적인 모델 재학습, 데이터 전처리, 배치 추론 등에 활용됩니다. Cron 표현식으로 실행 주기를 정의할 수 있습니다. |
| **스토리지** (Storage) | 애플리케이션의 데이터를 영구적으로 보관하기 위한 저장 공간을 관리하는 기능입니다. Runway는 PVC, Shared PVC 등 다양한 스토리지 옵션을 제공합니다. 데이터 손실 방지와 효율적인 용량 관리가 핵심입니다. |
| **스토리지 클래스** (Storage Class) | Kubernetes에서 볼륨이 어떻게 프로비저닝될지 정의하는 속성입니다. Volume 모드와 Filer 모드 등이 있으며, 각 모드마다 성능 특성과 사용 방법이 다릅니다. 애플리케이션의 요구사항에 맞는 스토리지 클래스를 선택해야 합니다. |
| **실시간 서빙** (Real-time Serving) | 사용자 요청을 받으면 즉시 모델 추론을 수행하고 응답을 반환하는 방식입니다. 낮은 레이턴시가 중요하며, 대화형 AI, 추천 시스템 등에 사용됩니다. 배치 서빙에 비해 응답 속도는 빠르지만 처리량은 낮을 수 있습니다. |

## ㅇ

| 용어 | 설명 |
|------|------|
| **아티팩트** (Artifact) | 모델 학습의 결과물로 생성된 파일과 메타데이터입니다. 학습된 가중치, 설정 파일, 메트릭, 로그 등을 포함합니다. MLflow 등을 통해 체계적으로 관리하며, 재현성과 추적성을 보장합니다. |
| **애플리케이션** (Application) | 사용자가 개발, 실험, 분석 등의 작업을 수행할 수 있는 실행 환경입니다. Jupyter Notebook, VS Code, RStudio 등 다양한 애플리케이션을 배포할 수 있습니다. 프로젝트 구성원 간 공유하거나 개인 전용으로 사용할 수 있습니다. |
| **앱 배포** (App Deployment) | 애플리케이션 템플릿을 선택하고 리소스와 설정을 지정하여 실행하는 과정입니다. CPU, 메모리, 스토리지 등을 설정하며, 배포 완료 후 즉시 접속하여 사용할 수 있습니다. No-Code 방식으로 간편하게 개발 환경을 구성합니다. |
| **앱 카탈로그** (App Catalog) | Runway에서 사전 구성된 애플리케이션 템플릿을 제공하는 메뉴입니다. Jupyter, VS Code, MLflow 등 다양한 도구를 클릭 몇 번으로 빠르게 배포할 수 있습니다. 복잡한 설정 없이 즉시 사용 가능한 환경을 제공합니다. |
| **양자화** (Quantization) | 모델의 가중치나 활성화 값의 정밀도를 낮춰 크기와 연산량을 줄이는 최적화 기법입니다. FP32를 INT8 또는 INT4로 변환하여 메모리 사용량과 추론 속도를 개선합니다. 정확도 손실을 최소화하면서 효율성을 높이는 것이 핵심입니다. |
| **엔드포인트** (Endpoint) | 외부 시스템이 배포된 모델에 접근하기 위한 API 진입점입니다. REST API 형태로 제공되며, 하나의 엔드포인트에 여러 모델 배포를 구성할 수 있습니다. 트래픽 분배, A/B 테스트, 섀도우 테스트 등의 기능을 지원합니다. |
| **엔드포인트 URL** (Endpoint URL) | 외부에서 배포된 모델에 접근할 수 있는 전체 웹 주소입니다. 일반적으로 https://엔드포인트명.프로젝트.워크스페이스.베이스도메인 형식입니다. 이 URL로 REST API 요청을 보내 모델 추론을 수행합니다. |
| **엣지 디바이스** (Edge Device) | 네트워크의 가장자리(edge)에서 동작하는 컴퓨팅 장치입니다. 스마트폰, IoT 센서, 엣지 서버 등이 해당되며, 클라우드로 데이터를 전송하지 않고 로컬에서 처리합니다. 낮은 레이턴시와 프라이버시 보호가 장점입니다. |
| **역할** (Role) | 사용자에게 부여할 권한 집합을 정의한 그룹입니다. Admin, Member, Viewer 등 사전 정의된 역할과 사용자 정의 역할이 있습니다. 역할 기반 접근 제어(RBAC)를 통해 보안과 권한 관리를 체계적으로 수행합니다. |
| **오케스트레이션** (Orchestration) | 여러 컨테이너의 배포, 확장, 네트워킹, 관리를 자동화하는 프로세스입니다. Kubernetes가 대표적인 컨테이너 오케스트레이션 플랫폼입니다. 수천 개의 컨테이너를 효율적으로 운영할 수 있게 합니다. |
| **오토 스케일링** (Auto Scaling) | 트래픽이나 리소스 사용량에 따라 워크로드 인스턴스 수를 자동으로 증감하는 기능입니다. 수평 확장(인스턴스 추가/제거)과 수직 확장(리소스 증가/감소)이 가능합니다. 효율적인 리소스 활용과 안정적인 서비스 제공을 동시에 달성합니다. |
| **온프레미스** (On-premises) | 클라우드가 아닌 조직의 자체 데이터센터에 구축된 인프라 환경입니다. 데이터 주권, 보안, 규정 준수 등의 이유로 선택됩니다. 클라우드에 비해 초기 투자가 크지만 장기적으로는 비용 효율적일 수 있습니다. |
| **용량** (Capacity) | 클러스터나 워크스페이스에서 사용 가능한 전체 리소스의 양입니다. CPU 코어 수, 메모리 용량, GPU 수 등으로 표현됩니다. 용량 대비 사용량을 모니터링하여 리소스를 효율적으로 관리합니다. |
| **워크로드** (Workload) | 클러스터에서 실행되는 작업 또는 애플리케이션의 배포 단위입니다. Pod, Deployment, Job 등 다양한 형태로 존재하며, 각각 특정 목적과 생명주기를 가집니다. 워크로드 관리는 Kubernetes의 핵심 개념입니다. |
| **워크스페이스** (Workspace) | Runway에서 여러 프로젝트를 관리하는 최상위 조직 단위입니다. 팀 단위로 리소스를 격리하고 독립적인 작업 환경을 제공합니다. 워크스페이스별로 리소스 할당량, 멤버, 권한을 관리할 수 있습니다. |
| **워크스페이스 역할** (Workspace Role) | 워크스페이스 내에서 사용자의 권한을 정의하는 역할입니다. 워크스페이스 설정, 프로젝트 생성, 멤버 관리 등의 권한을 제어합니다. 프로젝트 역할과는 별개로 관리됩니다. |
| **워크플로우** (Workflow) | 데이터 수집, 전처리, 학습, 평가, 배포 등 일련의 작업을 순차적으로 연결한 실행 흐름입니다. Apache Airflow 등으로 워크플로우를 정의하고 자동화할 수 있습니다. ML 파이프라인 구축의 기본 개념입니다. |
| **유사도 검색** (Similarity Search) | 벡터 공간에서 입력 벡터와 가장 유사한 벡터들을 찾는 검색 작업입니다. 코사인 유사도나 유클리드 거리 같은 메트릭으로 거리를 계산합니다. 의미 기반 검색과 추천 시스템의 핵심 기술입니다. |
| **인스턴스** (Instance) | 애플리케이션이나 모델의 실행 중인 복제본입니다. 트래픽 증가에 대응하기 위해 여러 인스턴스를 동시에 실행하여 부하를 분산할 수 있습니다. 오토 스케일링을 통해 인스턴스 수를 자동으로 조정합니다. |
| **인프라 코드화** (Infrastructure as Code) | 인프라 설정을 코드로 정의하고 버전 관리하는 방식입니다. YAML이나 JSON 파일로 서버, 네트워크, 스토리지 구성을 선언하고 자동으로 프로비저닝합니다. 재현 가능하고 일관된 인프라 환경을 제공합니다. |
| **임계값** (Threshold) | 특정 동작이나 알림을 트리거하는 기준이 되는 값입니다. CPU 사용률 80% 초과 시 알림, 메모리 부족 시 오토 스케일링 등의 규칙을 정의합니다. 임계값 설정은 안정적인 시스템 운영의 핵심입니다. |

| 용어 | 설명 |
|------|------|
| **재현성** (Reproducibility) | 동일한 데이터와 코드로 실험을 재실행했을 때 같은 결과를 얻을 수 있는 특성입니다. 랜덤 시드 고정, 환경 버전 관리, 데이터 버전 관리 등을 통해 재현성을 확보합니다. 과학적인 ML 연구와 디버깅에 필수적입니다. |
| **점진적 롤아웃** (Gradual Rollout) | 새로운 모델의 트래픽 비율을 시간에 따라 점진적으로 증가시키는 배포 전략입니다. 0% → 25% → 50% → 100% 식으로 단계적으로 트래픽을 이동하여 위험을 분산합니다. 각 단계에서 성능을 모니터링하며 안전하게 배포할 수 있습니다. |
| **접근 모드** (Access Mode) | Kubernetes에서 노드나 Pod가 스토리지에 접근하는 방식을 정의하는 옵션입니다. ReadWriteOnce(단일 노드 읽기/쓰기), ReadWriteMany(다중 노드 읽기/쓰기), ReadOnlyMany(다중 노드 읽기 전용) 등이 있습니다. 워크로드 특성에 맞는 접근 모드 선택이 중요합니다. |
| **지정률** (Assignment Ratio) | 전체 클러스터 용량 대비 워크스페이스나 프로젝트에 지정(할당)된 리소스의 비율입니다. 관리자는 지정률을 통해 팀별 리소스를 계획하고 배분합니다. 할당률과 함께 리소스 관리의 핵심 지표입니다. |

## ㅊ

| 용어 | 설명 |
|------|------|
| **처리량** (Throughput) | 단위 시간당 시스템이 처리할 수 있는 작업의 양입니다. 초당 요청 수(RPS), 시간당 처리 건수 등으로 측정됩니다. 높은 처리량은 많은 사용자를 동시에 처리할 수 있음을 의미합니다. |
| **추론** (Inference) | 학습이 완료된 모델에 새로운 입력 데이터를 제공하여 예측값을 얻는 과정입니다. 학습과 달리 모델 가중치를 업데이트하지 않으며, 일반적으로 학습보다 계산량이 적습니다. 프로덕션 환경에서는 추론 성능 최적화가 중요합니다. |
| **추론 URL** (Inference URL) | 모델 추론을 위한 REST API 엔드포인트의 구체적인 경로입니다. V2 프로토콜의 경우 /v2/models/{모델명}/infer 형식을 사용합니다. 이 경로로 POST 요청을 보내 입력 데이터를 전송하고 예측 결과를 받습니다. |
| **추적성** (Traceability) | 데이터, 모델, 실험의 변경 이력을 기록하고 추적할 수 있는 특성입니다. 어떤 데이터로 어떤 모델을 학습했는지, 하이퍼파라미터는 무엇이었는지 등을 추적합니다. 거버넌스와 재현성을 위한 핵심 요소입니다. |

## ㅋ

| 용어 | 설명 |
|------|------|
| **카나리 배포** (Canary Deployment) | 새로운 모델 버전을 소량의 트래픽(일반적으로 10%)에만 먼저 배포하여 안정성을 검증한 후 점진적으로 비율을 증가시키는 배포 전략입니다. 문제 발생 시 빠르게 롤백할 수 있어 위험을 최소화합니다. |
| **컨테이너** (Container) | 애플리케이션과 모든 의존성을 하나의 패키지로 묶은 경량 실행 단위입니다. 호스트 OS를 공유하므로 VM보다 가볍고 빠르게 시작됩니다. Docker가 가장 널리 사용되는 컨테이너 기술입니다. |
| **컴퓨팅 리소스** (Computing Resources) | 워크로드 실행에 필요한 CPU, 메모리, GPU 등의 하드웨어 자원입니다. Runway에서는 애플리케이션이나 모델 배포 시 필요한 리소스를 지정합니다. 적절한 리소스 할당이 성능과 비용 효율성을 좌우합니다. |

| **MLflow** | 머신러닝 실험, 모델, 배포를 관리하기 위한 오픈소스 플랫폼입니다. 실험 추적, 모델 레지스트리, 배포 기능을 제공합니다. 프레임워크에 구애받지 않아 다양한 ML 프로젝트에서 활용됩니다. |
| **MiB** | 메비바이트(Mebibyte)로, 2^20 바이트(1,048,576 바이트)를 나타내는 단위입니다. MB(메가바이트, 10^6 바이트)와는 약간 다른 이진 단위입니다. 메모리 용량 표시에 주로 사용됩니다. |

## N

| 용어 | 설명 |

| **No Code** | 코드를 전혀 작성하지 않고 시각적 인터페이스만으로 애플리케이션을 구축하는 방식입니다. 드래그 앤 드롭, 폼 입력 등으로 모든 설정을 완료합니다. 비개발자도 애플리케이션을 빠르게 생성할 수 있습니다. |

## O

| 용어 | 설명 |

| **ONNX** | 다양한 ML 프레임워크 간 모델 호환을 위한 개방형 포맷입니다. PyTorch, TensorFlow 등에서 학습한 모델을 ONNX로 변환하여 다른 런타임에서 실행할 수 있습니다. 모델 배포와 최적화를 용이하게 합니다. |

## P

| 용어 | 설명 |
|------|------|
| **PVC** | Persistent Volume Claim의 약자로, Kubernetes에서 영구 저장 공간을 요청하는 리소스입니다. 단일 Pod만 마운트 가능한 전용 볼륨을 제공합니다. 데이터베이스나 상태를 유지해야 하는 애플리케이션에 필수적입니다. |
| **Pod** | Kubernetes에서 배포할 수 있는 가장 작은 단위로, 하나 이상의 컨테이너를 포함합니다. Pod 내 컨테이너는 네트워크와 스토리지를 공유합니다. Kubernetes의 기본 실행 단위입니다. |
| **Pro Code** | 전문적인 코드 작성을 통해 세밀한 제어와 커스터마이징을 수행하는 개발 방식입니다. Low Code/No Code에 비해 유연성이 높지만 전문 지식이 필요합니다. 복잡한 요구사항이나 성능 최적화에 적합합니다. |
| **PyTorch** | Facebook(Meta)이 개발한 오픈소스 딥러닝 프레임워크입니다. 동적 계산 그래프와 직관적인 API로 연구자들에게 인기가 높습니다. TensorFlow와 함께 가장 널리 사용되는 ML 프레임워크입니다. |
| **pickle** (Pickle) | Python 객체를 바이트 스트림으로 직렬화(serialize)하는 모듈과 포맷입니다. 모델, 데이터, 설정 등을 파일로 저장하고 로드할 때 사용합니다. Python 생태계에서 널리 사용되지만 보안 이슈에 주의해야 합니다. |

## R

| 용어 | 설명 |
|------|------|
| **RAG** (Retrieval-Augmented Generation) | 외부 지식 베이스에서 관련 정보를 검색하여 LLM의 응답 품질을 개선하는 기법입니다. 벡터 데이터베이스에서 유사한 문서를 찾아 프롬프트에 포함시켜 더 정확하고 최신의 답변을 생성합니다. 환각(hallucination) 문제를 완화하는 데 효과적입니다. |
| **REST API** | HTTP 프로토콜을 기반으로 하는 웹 서비스 API 아키텍처 스타일입니다. GET, POST, PUT, DELETE 등의 HTTP 메서드로 리소스를 조작합니다. 단순성과 확장성으로 인해 웹 서비스의 표준으로 자리 잡았습니다. |
| **ReadOnlyMany** | 여러 노드가 동시에 읽기만 수행할 수 있는 볼륨 접근 모드입니다. 공유 설정 파일이나 정적 데이터를 여러 Pod에서 읽어야 할 때 사용합니다. 데이터 일관성을 보장하면서 다중 접근을 허용합니다. |
| **ReadWriteMany** | 여러 노드가 동시에 읽기와 쓰기를 수행할 수 있는 볼륨 접근 모드입니다. NFS 같은 공유 파일 시스템이 필요하며, 다중 인스턴스가 동일한 데이터를 수정하는 경우에 사용합니다. 복잡한 동기화 문제를 고려해야 합니다. |
| **ReadWriteOnce** | Kubernetes 볼륨 접근 모드 중 하나로, 단일 노드가 읽기와 쓰기를 모두 수행할 수 있습니다. 가장 일반적인 접근 모드이며, 대부분의 스토리지 유형에서 지원됩니다. 단일 인스턴스 애플리케이션에 적합합니다. |
| **Runway** | AI 모델과 애플리케이션을 안정적으로 배포, 실행, 운영할 수 있는 통합 플랫폼입니다. ML 라이프사이클 전체를 지원하며, No-Code부터 Pro-Code까지 다양한 사용자 수준을 고려합니다. 온프레미스와 클라우드 환경 모두에서 운영 가능합니다. |

## S

| 용어 | 설명 |
|------|------|
| **SeaweedFS** | Kubernetes 환경에 최적화된 분산 파일 시스템입니다. 객체 스토리지와 POSIX 파일 시스템 인터페이스를 모두 지원합니다. 대용량 파일 저장과 빠른 읽기/쓰기 성능을 제공합니다. |
| **Shared PVC** | 여러 애플리케이션(Pod)에서 동시에 마운트하여 사용할 수 있는 공유 볼륨입니다. ReadWriteMany 접근 모드를 지원하며, 팀원 간 데이터 공유나 협업에 활용됩니다. 일반 PVC와 달리 다중 접근을 허용합니다. |
| **safetensors** | Hugging Face에서 개발한 안전하고 빠른 모델 가중치 저장 포맷입니다. pickle의 보안 취약점을 해결하고 로딩 속도를 개선했습니다. Hugging Face 모델 허브의 표준 포맷으로 채택되고 있습니다. |
| **sentence-transformers** (Sentence Transformers) | 문장 단위의 임베딩을 생성하기 위한 Python 프레임워크입니다. BERT, RoBERTa 등의 모델을 파인튜닝하여 의미론적으로 유사한 문장이 가까운 벡터로 표현되도록 학습합니다. 검색, 클러스터링, 의미 유사도 측정에 활용됩니다. |

## T
