# [Skill] Runway Kubeconfig 활용 및 플랫폼 관리자 가이드

본 문서는 Runway 플랫폼 외부에서 개발 컴퓨터의 CLI(Command Line Interface)나 CI/CD 파이프라인을 연계해 클러스터를 직접 제어하고, 워크스페이스 및 프로젝트를 관리하는 운영자/개발자용 고급 관리 가이드입니다.

---

## 1. Kubeconfig 다운로드 및 환경 설정 (Kubeconfig Utilization)

Kubeconfig는 외부 로컬 환경의 `kubectl` CLI 클라이언트가 Runway Kubernetes 클러스터에 보안 연결 및 API 명령을 요청할 수 있도록 인증 정보를 담고 있는 핵심 설정 파일입니다.

### 1) 다운로드 경로
- **메뉴 경로**: 오른쪽 상단 **사용자 프로필 아이콘** 클릭 > **내 계정** > **Kubeconfig** 탭 이동 > **다운로드** 버튼 클릭.

### 2) 클라이언트 환경 변수 설정
다운로드한 `kubeconfig.yaml` 파일을 터미널에서 사용할 수 있도록 활성화합니다.

#### Windows (PowerShell)
```powershell
# 특정 터미널 세션에 일시 적용
$env:KUBECONFIG="C:\Users\shain\Downloads\kubeconfig.yaml"

# 또는 기본 경로에 복사하여 영구 적용 (권장)
New-Item -ItemType Directory -Force -Path "$HOME\.kube"
Copy-Item -Path "C:\Users\shain\Downloads\kubeconfig.yaml" -Destination "$HOME\.kube\config" -Force
```

#### Linux / macOS / WSL (Bash)
```bash
# 영구 적용 설정 (.bashrc 또는 .zshrc 하단에 추가)
mkdir -p ~/.kube
cp ~/Downloads/kubeconfig.yaml ~/.kube/config
chmod 600 ~/.kube/config

# 또는 환경 변수로 임시 연동
export KUBECONFIG=~/.kube/config
```

### 3) 권한과 RBAC 제약
- Kubeconfig를 통해 수행할 수 있는 모든 API 명령 범위는 사용자의 **Runway 플랫폼 계정 역할(RBAC)**과 동일하게 제한됩니다.
- 예를 들어, 프로젝트의 **Project Member** 역할을 부여받은 사용자의 Kubeconfig로는 해당 프로젝트 네임스페이스 내에서만 Pod 조회, 배포, 삭제가 가능하며, 클러스터 전체 노드 정보 조회나 타 프로젝트 제어는 불가능합니다.

## 2. API 액세스 키 관리 (Access Keys)

Runway 외부 애플리케이션이나 자동화 파이프라인(GitHub Actions, Airflow 외부 러너 등)이 Runway OpenAPI를 호출할 수 있도록 고유 액세스 토큰을 발급받아 관리합니다.

- **메뉴 경로**: 오른쪽 상단 **내 계정** > **액세스 키** 탭.
- **키 생성**: **+ 생성** 버튼을 통해 고유 토큰 스트링을 신규 발급합니다.
- **안전 규칙**: 생성된 시점에 단 한 번만 복사가 가능하므로, 복사 후 환경변수 보관함에 즉시 안전하게 저장하십시오. 유출 즉시 **삭제(Delete)** 버튼을 클릭하여 파기해야 합니다.

## 3. 프로젝트 및 워크스페이스 관리 업무 (Administration)

워크스페이스 및 프로젝트 관리자가 멤버 관리와 자원 배정을 수행할 때 준수해야 할 행정 워크플로우 가이드라인입니다.

### 1) 프로젝트 멤버 및 역할 할당 (Project User Management)
- **메뉴 경로**: 프로젝트 > **설정** > **사용자 관리** 탭.
- **초대 규칙**: 워크스페이스에 등록되어 있는 사용자를 프로젝트 단위의 멤버로 검색하여 추가할 수 있습니다.
- **역할 지정**: 추가된 멤버에게 적절한 역할(`Admin`, `Member`, `Viewer`)을 부여하여 권한 오남용을 예방합니다.

### 2) 프로젝트 할당 리소스 수정 (Project Resource Modification)
프로젝트에 배정된 총 컴퓨팅 할당량(CPU, 메모리, GPU 등)을 클러스터 잔여 여유량 한도 내에서 상향 또는 하향 조정할 수 있습니다. (워크스페이스 어드민 권한 필요)
- **메뉴 경로**: 프로젝트 > **설정** > **리소스 변경** 탭.
- **제약**: 하향 조정 시, 프로젝트 내에서 현재 기동 중인 애플리케이션 및 모델 서빙 컨테이너의 실시간 사용량 합계 이하로는 줄일 수 없습니다. 자원이 차단되어 서비스 장애가 생기는 것을 스케줄러가 방지하기 위함입니다.

### 3) 프로젝트 삭제 (Project Deletion)
- **메뉴 경로**: 프로젝트 > **설정** > **위험 구역(Danger Zone)** > **프로젝트 삭제** 버튼 클릭.
- **중요 영향성**: 프로젝트 삭제 시 프로젝트 하위의 **모든 애플리케이션, 추론 엔드포인트, 배포 모델, 그리고 스토리지 볼륨이 완전히 영구 소멸**됩니다. 백업 데이터가 있는지 최종 확인하고 신중히 결정해야 합니다.

### 4) 워크스페이스 신규 프로젝트 생성 (Project Creation in Workspace)
- **메뉴 경로**: 워크스페이스 상세 화면 > 오른쪽 상단 **+ 프로젝트 생성** 버튼 클릭.
- **기본 입력**: 프로젝트 명칭, ID, 배정할 초기 쿼터(Quota - CPU, 메모리, GPU 등)를 기입합니다.

## 4. 에이전트 가이드라인 (Agent Best Practices)

- **Kubectl 명령어 보안 필터**: 에이전트가 Kubeconfig 설정을 이용한 CLI 작업을 지원할 시, 사용자의 권한 범위를 넘어서는 `kubectl` 명령이나 보안에 민감한 네임스페이스 격리 유출 명령을 제어하도록 조언하고 안내하십시오.
- **프로젝트 삭제 위험 경고**: 사용자가 프로젝트나 대형 볼륨 삭제를 요청하거나 계획할 시, 에이전트는 **"하위 애플리케이션 및 영구 데이터가 전부 소멸되어 복구 불가능하다"**는 경고와 백업의 필요성을 반드시 붉은색 경고(alert) 등을 포함해 안내해야 합니다.
- **리소스 쿼터 모니터링**: 프로젝트 리소스 부족 이슈 발생 시, 단순 할당 에러 메시지를 노출하지 말고, 워크스페이스 어드민을 통해 **프로젝트 쿼터 변경** 탭에서 한도를 조정하거나 사용하지 않는 애플리케이션을 배포 해제(Undeploy)하여 리소스를 확보하도록 안내하십시오.
