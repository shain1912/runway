# [Skill] Runway 개발 환경 구축 및 애플리케이션 생성 가이드

본 문서는 Runway에서 코드를 작성하고 모델을 실험할 수 있는 개발 환경을 구축하고 애플리케이션을 배포하는 구체적인 워크플로우에 대한 가이드입니다.

---

## 1. 첫 개발 환경 구축 3단계 워크플로우

프로젝트에 할당받은 후 개발을 시작하기 위한 핵심 표준 절차입니다.

```
+--------------------------+
|  1. 스토리지 볼륨 생성   |  --> 코딩 파일과 데이터를 보관할 영구 저장 공간 확보
+--------------------------+
             |
             v
+--------------------------+
|   2. 개발용 앱 배포      |  --> 앱 카탈로그에서 Code Server 또는 JupyterLab 선택 후 배포
+--------------------------+
             |
             v
+--------------------------+
|    3. 웹 브라우저 접속   |  --> 생성된 httpRoute 도메인 링크로 접속하여 개발 시작
+--------------------------+
```

---

## 2. 1단계: 스토리지 볼륨 생성 및 관리

애플리케이션이 삭제되거나 재시작되어도 작업한 소스코드와 데이터가 손실되지 않도록 **영구 볼륨(Persistent Volume Claim)**을 사전에 생성해야 합니다.

### 볼륨 설정 핵심 옵션

| 항목 | 선택 옵션 및 기준 | 설명 |
|---|---|---|
| **스토리지 클래스<br>(Storage Class)** | `ceph-block` (권장) | 단일 Pod 전용 고성능 블록 스토리지. 개인 소스코드 작업 공간에 최적화. |
| | `ceph-filesystem` | 다중 컨테이너 및 공유형 분산 파일시스템. |
| **접근 모드<br>(Access Mode)** | `ReadWriteOnce` (RWO) | 단일 애플리케이션(Pod)에서만 마운트하여 읽기/쓰기 가능. `ceph-block` 선택 시 필수. |
| | `ReadWriteMany` (RWX) | 여러 애플리케이션에서 동시에 마운트하여 데이터 공유 가능. `ceph-filesystem` 선택 시 사용. |
| **용량 (Size)** | 예: `10 GiB` ~ `100 GiB` | 데이터 크기 및 코드 용량을 산정하여 기입. |

### 생성 경로 및 제약
- **UI 경로**: 프로젝트 > **스토리지** > 오른쪽 상단 **+ 생성** 버튼 클릭.
- **볼륨 ID 제약**: 3~63자의 영문 소문자, 숫자, 하이픈(-)만 사용 가능.

---

## 3. 2단계: 애플리케이션(앱) 배포 및 분류

Runway는 AI/ML 워크로드를 패키징한 실행 환경을 **애플리케이션(App)** 단위로 정의합니다.

### 1) 애플리케이션 유형

| 구분 | 배포 방식 | 제공 목적 및 예시 |
|---|---|---|
| **플랫폼 앱** | 자동 설치 | 워크스페이스/프로젝트 활성화 시 플랫폼이 기본 배포하는 공유 인프라.<br>- 예: **Gitea**(형상관리), **MLflow**(실험 추적), **Airflow**(워크플로우), **Argo CD**(GitOps 배포) |
| **카탈로그 앱** | 앱 카탈로그 | 검증된 템플릿(Helm Chart) 목록에서 선택해 최소 설정만으로 즉시 설치.<br>- 예: **Code Server**(VS Code 웹 버전), **JupyterLab**, **Vector DB**(Qdrant 등) |
| **커스텀 앱** | 직접 배포 | 사용자가 보유한 임의의 Helm 차트 및 values.yaml을 직접 업로드하거나 기입하여 배포. 높은 확장성 제공. |

---

## 4. values.yaml 핵심 설정 템플릿 (Code Server/JupyterLab 기준)

카탈로그에서 애플리케이션을 생성할 때, `values.yaml` 에디터에서 볼륨 연동 및 외부 접근 주소 설정을 지정해야 정상 작동합니다.

### 1) 볼륨 연동 (`persistence`)
미리 생성해 둔 스토리지 볼륨의 ID를 연결하여 개발 파일을 영구 보존합니다.

```yaml
persistence:
  enabled: true
  mountPath: /config              # 컨테이너 내 마운트 경로 (Code Server는 /config 권장)
  existingClaim: energy-data-vol  # 1단계에서 생성한 스토리지 볼륨 ID 기입 (주석 해제 필수)
```

### 2) 브라우저 접속을 위한 라우팅 설정 (`httpRoute`)
도메인네임을 활성화하여 개발 도구 웹 UI에 접속합니다.

```yaml
httpRoute:
  enabled: true                               # false에서 true로 수정 필수
  hostname: "csef0227.v2.mrxrunway.ai"        # {원하는서브도메인}.{Runway베이스도메인}
```
> **Info**: 베이스 도메인 확인
> 리소스 현황 최상단 영역의 **베이스 도메인** 주소를 복사하여 hostname 뒤에 도메인 접미사로 붙여 넣습니다. 서브도메인은 애플리케이션 ID와 일치시키는 것을 강력히 권장합니다.

---

## 5. 3단계: 애플리케이션 접속 및 수명 주기 관리

### 애플리케이션 실행 제어 (Runtime Control)
배포된 앱 카드를 클릭하면 상세 수명 주기를 제어할 수 있습니다.
- **배포 (Deploy)**: 앱을 클러스터에 배포하고 리소스를 활성화합니다.
- **배포 해제 (Undeploy)**: 컨테이너 실행을 중지하고 리소스를 반환하지만, 설정(values.yaml)은 그대로 유지됩니다. (자원 절약 시 유용)
- **수정 (Edit)**: `values.yaml`이나 메타데이터를 수정하고 다시 롤아웃합니다.
- **삭제 (Delete)**: 앱 설정과 리소스를 완전히 영구 삭제합니다. (연결된 `existingClaim` 볼륨은 삭제되지 않고 안전하게 보존됨)

## 6. 에이전트 가이드라인 (Agent Best Practices)

- **볼륨 사전 검증**: 애플리케이션 배포 및 수정 시 `values.yaml` 내 `existingClaim` 필드에 기입된 볼륨 ID가 실제로 프로젝트 내에 존재하고 상태가 `Bound` 인지 사전에 점검하십시오.
- **httpRoute 중복 방지**: 동일 프로젝트 내에서 두 개 이상의 애플리케이션이 동일한 `hostname`을 점유할 수 없습니다. 에이전트는 사용자가 제시한 서브도메인이 유일한지 검토하거나 고유 ID를 서브도메인으로 적용하도록 권장하십시오.
- **Argo CD 모니터링 연계**: 앱 상태가 정상적으로 배포(Healthy) 상태로 도달하지 않는 경우, 사용자에게 상세 디버깅을 위해 UI 상의 **Argo CD URL** 링크를 참조하도록 즉시 안내하십시오.
