# [Skill] Runway 소개 및 계정/역할 설정 가이드

본 문서는 Runway 2.0 플랫폼의 핵심 개념, 계층 구조, 그리고 사용자 역할 및 접근 범위에 대한 가이드입니다. 에이전트가 Runway 환경을 이해하고 사용자의 계정 및 역할을 관리할 때 참조합니다.

---

## 1. Runway 플랫폼 개요

Runway는 표준화된 운영 체계 위에서 AI·ML 워크로드를 **일관된 방식으로 구성·배포·운영**할 수 있도록 설계된 **AI 운영체제(AI OS)**입니다.

### 핵심 가치 및 특징
- **통합 운영 모델**: 자원 관리, 인증·인가, 배포, 모니터링(Observability)을 플랫폼 계층에서 통합 처리하여 인프라 복잡도를 숨김.
- **모듈형 구조**: 프로젝트 요구사항에 맞춰 필요한 도구만 선택하여 구성 가능.
- **책임의 분리**: 사용자는 모델 개발 및 실험 자체에 집중하고, 보안 및 인프라 제어는 플랫폼이 담당.

---

## 2. 작업 단위와 계층 구조 (Workspace & Project)

Runway는 멀티 테넌시 환경을 제공하기 위해 **워크스페이스(Workspace)**와 **프로젝트(Project)**라는 두 가지 논리적인 작업 단위를 기준으로 한 계층적 구조를 가집니다.

```
+-------------------------------------------------------------+
|                     Runway Platform                         |
+-------------------------------------------------------------+
                               |
                               v
+-------------------------------------------------------------+
|                      Workspace A                            |
|  - 사용자 그룹, 공통 정책 및 자원 범위 설정                |
+-------------------------------------------------------------+
             |                                   |
             v                                   v
+---------------------------+       +---------------------------+
|        Project A-1        |       |        Project A-2        |
|  - AI/ML 워크로드 실행    |       |  - AI/ML 워크로드 실행    |
|  - 스토리지 볼륨          |       |  - 스토리지 볼륨          |
|  - 애플리케이션 (앱)      |       |  - 애플리케이션 (앱)      |
|  - 추론 엔드포인트        |       |  - 추론 엔드포인트        |
+---------------------------+       +---------------------------+
```

### 1) 워크스페이스 (Workspace)
- **정의**: 여러 프로젝트를 논리적으로 묶어주는 상위 협업 환경.
- **용도**: 부서, 본부 또는 대규모 목적 단위로 자원을 배분하고 공통 정책을 적용하는 경계선.
- **포함 요소**: 워크스페이스 사용자 목록, 워크스페이스 역할, 하위 프로젝트 목록.

### 2) 프로젝트 (Project)
- **정의**: AI·ML 워크로드(개발 환경, 모델 서빙 등)가 실제로 실행되는 기본 작업 단위.
- **용도**: 개별 머신러닝 과제나 서비스 단위로 격리된 인프라(네임스페이스) 제공.
- **포함 요소**: 스토리지 볼륨, 애플리케이션(코드 서버, 주피터랩 등), 추론 엔드포인트, 프로젝트별 접근 권한 목록.

## 3. 로그인 및 인증 (Authentication)

Runway는 **Keycloak** 기반의 통합 인증 시스템(SSO)을 사용하여 모든 서비스에 대한 단일 로그인 및 강력한 인증 체계를 보장합니다.

### OIDC 및 OAuth 기반 인증 Flow
1. 사용자가 Runway Web UI에 접속하여 **로그인** 시도.
2. Keycloak 통합 로그인 페이지로 리다이렉트.
3. 아이디/패스워드 입력 후 인증 완료 시, JWT(JSON Web Token) 형식의 **Access Token** 및 **ID Token** 발급.
4. 이후 Runway API 호출 및 개별 애플리케이션(Code Server, JupyterLab 등) 접속 시 토큰 기반의 무인증 SSO 접근을 수행.

## 4. 사용자 역할 및 권한 체계 (Roles & Permissions)

Runway는 워크스페이스 수준과 프로젝트 수준에서 각각 별도의 역할(Role)을 부여하는 **역할 기반 접근 제어(RBAC)** 구조를 가집니다.

### 1) 워크스페이스 역할 (Workspace Roles)

워크스페이스 내에서의 사용자 관리, 설정 변경 및 프로젝트 생성 범위를 규정합니다.

| 역할 이름 | 권한 범위 설명 |
|---|---|
| **Workspace Owner** | 워크스페이스의 모든 자원과 설정에 대한 완전한 권한 보유. 워크스페이스 삭제, 일반 정보 변경, 사용자 초대 및 역할 변경 가능. |
| **Workspace Admin** | 워크스페이스 Owner에 준하는 권한을 가지나, 워크스페이스 삭제 및 Owner 지정/변경 권한은 제외됨. 프로젝트 생성/관리 가능. |
| **Workspace Member** | 워크스페이스 내 프로젝트를 조회할 수 있으며, 자신이 참여한 프로젝트 내의 작업 수행 가능. 신규 프로젝트 생성 권한 없음. |
| **Workspace Viewer** | 워크스페이스의 기본 정보 및 하위 프로젝트 목록만 읽기 전용으로 조회 가능. 변경 또는 실행 권한 없음. |

### 2) 프로젝트 역할 (Project Roles)

프로젝트 내부의 애플리케이션 배포, 스토리지 볼륨 생성, 모델 서빙 엔드포인트 관리 권한을 규정합니다.

| 역할 이름 | 권한 범위 설명 |
|---|---|
| **Project Admin** | 프로젝트 내의 모든 자원 관리 및 접근 권한 보유. 프로젝트 설정 수정, 사용자 추가/삭제 및 역할 지정, 볼륨/앱/엔드포인트 생성 및 삭제 가능. |
| **Project Member** | 실질적인 머신러닝 개발자 역할. 스토리지 볼륨 생성/수정, 애플리케이션 배포 및 제어, 모델 배포 및 트래픽 설정 변경 가능. 프로젝트 관리 설정은 불가. |
| **Project Viewer** | 프로젝트 내의 애플리케이션 상태, 볼륨 목록, 추론 엔드포인트 현황을 조회(Read-only)만 가능. 생성/수정/삭제 권한 없음. |

---

## 5. 에이전트 가이드라인 (Agent Best Practices)

- **역할 확인**: 사용자가 특정 애플리케이션 배포나 리소스 수정 명령을 내릴 때, 사용자 계정이 해당 프로젝트의 **Project Member** 또는 **Project Admin** 역할을 충족하는지 사전에 계정 정보(`e:\runway\docs_markdown\guide\manage\account\account-profile\index.md` 참고)를 통해 검증해야 합니다.
- **계층 구조 준수**: 모든 볼륨 생성 및 애플리케이션 배포 시에는 반드시 타겟 `Workspace ID`와 `Project ID`가 올바르게 지정되었는지 확인하여 테넌트 간 격리를 준수합니다.
