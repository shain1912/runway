# [Skill] Runway 모델 서빙 및 배포 전략 가이드

본 문서는 Runway 플랫폼에서 학습을 완료한 머신러닝/딥러닝 모델을 웹 서비스 환경에 배포하고, 점진적 롤아웃이나 A/B 테스팅 등의 정교한 트래픽 분배 전략을 수립하여 실시간 추론을 수행하는 전체 운영(MLOps) 단계의 가이드입니다.

---

## 1. 모델 서빙 2단계 프로세스

Runway에서의 모델 실시간 API 서비스 구축은 크게 **추론 엔드포인트 생성**과 **모델 배포** 단계로 구성됩니다.

```
+---------------------------------------+
|    1. 추론 엔드포인트(Endpoint) 생성  |  --> 도메인 및 런타임 유형(MLServer, Triton 등) 정의
+---------------------------------------+
                   |
                   v
+---------------------------------------+
|      2. 모델 배포(Model Deployment)   |  --> 스토리지 볼륨 내 모델 경로 로드, 리소스 및 트래픽 분배 지정
+---------------------------------------+
```

---

## 2. 1단계: 추론 엔드포인트 생성

- **메뉴 경로**: 프로젝트 > **추론 엔드포인트** > 오른쪽 상단 **+ 생성** 버튼 클릭.
- **주요 파라미터**:
  - **엔드포인트 ID**: 도메인 접두사로 활용되는 시스템 고유 식별자.
  - **서빙 런타임 (Serving Runtime)**: 배포 모델의 프레임워크에 최적화된 실행기 선택.
    - `MLServer`: Python, Scikit-Learn, XGBoost, LightGBM, Custom PyTorch 등에 적합.
    - `Triton Inference Server`: ONNX, TensorRT, PyTorch(TorchScript) 고속 병렬 추론에 최적화.
  - **접속 방식**: 외부 브라우저나 클라이언트 API에서 직접 접속하도록 `외부 공개` 설정을 지정합니다.

---

## 3. 2단계: 모델 배포 및 런타임별 디렉터리 경로 설정

배포할 엔드포인트 상세 내에서 **+ 모델 배포** 버튼을 클릭하여 모델 정보를 기입합니다. 현재 모델 저장소 소스는 **프로젝트 스토리지 볼륨**만 지원됩니다.

### ⚠️ [중요] 서빙 런타임별 모델 경로 매핑 규칙 차이

선택한 서빙 엔진에 따라 모델 파일이 저장된 스토리지 볼륨 내의 디렉터리 입력 및 배치 규칙이 완전히 상이하므로 주의해야 합니다.

```
[MLServer 경로 해석 방식]
지정한 볼륨 내 디렉터리를 '모델 파일이 직접 위치한 최종 경로'로 간주합니다.

  볼륨 내 저장 구조 예시:
    auto-mpg-model/       <-- 모델 배포 시 입력할 상대 경로
      ├── MLmodel         (MLServer 설정 메타파일)
      └── model.pkl       (실제 모델 가중치 파일)

  -> 모델 경로 입력값: "auto-mpg-model"


[Triton 경로 해석 방식]
지정한 볼륨 내 디렉터리를 'Triton Model Repository Root'로 해석하며, 그 하위에 정해진 트리 구조(버전 폴더 및 설정 파일)가 존재해야 합니다.

  볼륨 내 저장 구조 예시:
    / (볼륨 최상위)
      └── my-onnx-model/  (Triton이 인식할 공식 모델 이름 폴더)
            ├── config.pbtxt (Triton 설정 파일)
            └── 1/
                 └── model.onnx (버전 1 가중치 파일)

  -> 모델 경로 입력값: 빈 칸(볼륨 루트인 /mnt/models/가 Repository Root가 됨)
```

## 4. 컴퓨팅 리소스 및 스케일링 설정

- **컴퓨팅 자원**: CPU(millicores 단위 정수, 예: 1000 = 1Core), Memory(MiB 단위 정수, 최소 64MiB)를 모델 바이너리 크기와 연산 강도에 기반해 지정합니다.
- **GPU 가속**: 딥러닝/LLM 모델 추론 시 토글을 켜고 원하는 GPU 모델명과 코어/메모리 할당을 지정합니다. (공유 메모리 설정은 고급 옵션에서 `/dev/shm` 마운트 후 지정).
- **복제본 수 (Replicas)**: 트래픽 동시 처리량을 확보하기 위해 실행할 모델 인스턴스 복제 개수(1~100개)를 설정합니다.

---

## 5. 트래픽 가중치 분배 및 배포 운영 전략

Runway는 상대적 가중치(Relative Weight) 방식을 사용하여 여러 개의 모델 버전을 하나의 추론 엔드포인트 뒤에 동시 배치하고 트래픽을 정밀하게 통제합니다.

### 1) 가중치 비율 계산 공식
$$\text{모델 분배 비율 (\%)} = \left( \frac{\text{해당 모델 가중치}}{\text{전체 모델 가중치 합}} \right) \times 100$$

### 2) 배포 전략 시나리오별 설정 예시

| 전략 | 설명 | 가중치 설정 예시 |
|---|---|---|
| **A/B 테스팅** | 두 가지 아키텍처 모델을 1:1 대조군 평가를 통해 성능 검증. | 기존모델: `1` (50%)<br>신규모델: `1` (50%) |
| **카나리 배포** | 일부 유입 트래픽에만 신규 버전을 소량 흘려 오동작 여부 모니터링. | 프로덕션: `9` (90%)<br>카나리모델: `1` (10%) |
| **점진적 롤아웃** | 점차 신버전 트래픽 비율을 늘려 무중단 교체 수행. | 1단계: 기존 `7` / 신규 `3`<br>2단계: 기존 `5` / 신규 `5`<br>3단계: 기존 `0` / 신규 `10` (교체 완료) |
| **섀도우 테스트** | 프로덕션에는 영향을 주지 않고 신버전 출력을 수집. | 기존모델: `1` (100% 트래픽 실서비스)<br>섀도우: `0` (트래픽 라우팅은 없으나 비동기 로깅 수집) |

### 3) 유효 가중치 (Effective Weight) 개념
- 사용자가 설정한 값(Target Weight)과 별개로, **실제 트래픽이 라우팅되는 정규화된 최종 비율**입니다.
- 특정 배포 모델이 리소스 준비 중이거나 에러 상태(Unhealthy)인 경우, 분배 라우팅 대상에서 제외되며 자동으로 **정상(Healthy)인 모델들만 추출해 100%로 재계산한 유효 가중치**가 실시간 적용되어 중단 없는 가용성을 확보합니다.

## 6. 에이전트 가이드라인 (Agent Best Practices)

- **런타임 경로 오류 예방**: 에이전트는 사용자가 MLServer 또는 Triton 런타임을 타겟팅할 때, 볼륨의 저장 경로 트리 구조가 각각의 디렉터리 해석 매핑 방식(MLServer = 모델 포함 폴더 경로 직접 지정 vs Triton = 버전/설정이 들어있는 부모 폴더를 Repository Root로 하여 지정)과 완벽히 호환되는지 사전에 논리적 검토를 수행하십시오.
- **가중치 유효성 확인**: 트래픽 편집 시 모든 모델 배포의 가중치를 동시에 `0`으로 지정하는 설정을 방지하고, 점진적 롤아웃 시 각 단계별로 가중치의 합산 분할이 타겟 비율과 맞는지 계산 검증을 도와주십시오.
- **Argo CD 모니터링 연동**: 모델 배포 상태가 `Unhealthy` 이거나 실행이 지연될 시 상세 Pod 기동 로그를 확인하도록 사용자에게 엔드포인트 내 상세 화면의 **Argo CD** 링크를 신속하게 제공하도록 하십시오.
